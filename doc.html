Hybrid Project Documentation

A comprehensive guide to the eSewa Hybrid Mobile Application, documenting how the native Android and iOS applications integrate with a shared Flutter module.

Project Goal: Build a high-performance, secure hybrid app where sensitive logic (Auth/Session) stays native, while complex UI code (Product List/Detail) is shared via Flutter.

Exact Requirements (project.pdf)

The following exact requirements were extracted from the project specification document:

  • ✓

    Native Platforms

    Develop using Kotlin (Android SDK 35) and Swift (Latest) for iOS.

  • ✓

    Dynamic Theming

    Pass Light/Dark mode, Colors, Spacing, and Typography from Native to Flutter at runtime.

  • ✓

    Navigation & Session

    Generate & cache random UUID on login. Bypass login if cached UUID exists. Use UUID in all API Headers.

  • ✓

    Flutter Module Features

    Implemented using Riverpod, Dio, and fakeStoreApi. Must return selected product payload to native on "Pay".

Android Page Walkthrough

1

App.kt (The Engine Hub)

The core of the Android implementation. It "pre-warms" the Flutter Engine so that transitions into the Flutter module are instantaneous. It also contains buildAppConfig() which gathers all theme tokens from encrypted storage.

2

MainActivity.kt (The Bridge)

This class hosts the FlutterActivity. It sets up the MethodChannel named com.app.native/flutter. It listens for requests from Dart and delegates actions like Payment or Auth status back to native managers.

3

LoginActivity.kt (Auth Entry)

A native Jetpack Compose screen. It handles user registration and login. It uses SHA-256 for password hashing and stores credentials in EncryptedSharedPreferences for maximum security.

iOS Page Walkthrough

1

FlutterManager.swift (Engine lifecycle)

Acts as the bridge between SwiftUI and Flutter. It manages the FlutterEngine and the FlutterMethodChannel. It ensures that whenever the app state changes (like theme toggle), the Flutter module is updated via refreshConfig().

2

LoginView.swift (Native UI)

A SwiftUI based login screen. It follows the same flow as Android: checks Keychain for existing UUID, generates new session data on success, and transitions to the main view.

Flutter Page Walkthrough

1

main.dart (Hybrid Entry)

Uses @pragma('vm:entry-point') to define mainWithConfig. This allows the native platform to start the module while passing the entire configuration JSON directly into Riverpod's initialConfigProvider.

2

product_list_screen.dart (The Catalog)

Uses productListProvider to fetch data from fakestoreapi. It displays a clean, responsive list using the spacing and color tokens received from the native host.

3

api_client.dart (The Networker)

An advanced Dio client implementation. It features Offline Caching (saving JSON to SharedPreferences) and Secure Logging (stripping sensitive data in production).
