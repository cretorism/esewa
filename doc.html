Hybrid Native + Flutter Architecture

Welcome to the implementation documentation for the eSewa Hybrid Subscrition App. This project demonstrates a high-performance integration where native Android and iOS applications host a shared Flutter module.

Why Hybrid?
It allows us to keep sensitive logic like authentication and security in native code while sharing complex UI components like product lists and detail pages across platforms using Flutter.
Android SDK 35 iOS 12.0+ Flutter 3.x

Requirement Solving

This project was designed to solve several key business and technical requirements:

1

Secure Native Authentication

Requirement: Ensure user authentication is handled natively for better security integration.

Solution: Both Android and iOS implement a native LoginActivity and LoginView. Only after a successful login is the unique session UUID generated and passed to the Flutter module.

2

Unified Product Catalog

Requirement: Maintain a single UI codebase for complex product browsing and detail views.

Solution: The flutter_product_module contains all the Dart code for product listing and detail screens. This ensures a consistent user experience on both platforms with zero UI duplication.

3

Dynamic Theming

Requirement: The Flutter module must adapt to the design system dictated by the native app.

Solution: Upon initialization, the native layer sends an AppConfig object via MethodChannel. This contains colors, spacing, and typography tokens which Flutter uses to build its theme dynamically.

Phase 1: Flutter Module Development

The core of the application logic resides in the Flutter module. It uses Riverpod for state management and Dio for networking.

How the Flutter Entry Point Works

The module supports two entry points: a default one for debugging and a mainWithConfig for native integration.

// lib/main.dart
@pragma('vm:entry-point')
void mainWithConfig(Map<String, dynamic> initialConfig) {
  runApp(FlutterModuleApp(initialConfig: initialConfig));
}

class FlutterModuleApp extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    return ProviderScope(
      overrides: [
        if (initialConfig != null)
          initialConfigProvider.overrideWithValue(initialConfig),
      ],
      child: MaterialApp(home: ProductListScreen()),
    );
  }
}

Logic: The ProviderScope override allows us to inject the native configuration immediately, making it available throughout the widget tree before any screen is rendered.

Product List Screen

Displays a grid of products fetched from FakeStoreAPI. It uses productListProvider to handle the asynchronous state (Loading, Data, Error).

Phase 2: Android Native Implementation

The Android side is built with Kotlin and uses a "Pre-warmed" Flutter Engine for instant transitions.

The Pre-warmed Engine

In the App.kt class, we initialize the Flutter engine as soon as the application starts.

// App.kt
flutterEngine = FlutterEngine(this).apply {
    dartExecutor.executeDartEntrypoint(
        DartExecutor.DartEntrypoint.createDefault()
    )
}
// Cache it for later use
FlutterEngineCache.getInstance().put(FLUTTER_ENGINE_ID, flutterEngine)

Benefit: This eliminates the 1-2 second "white screen" or delay that usually occurs when starting a Flutter module, as the VM is already running in the background.

Passing Data to Flutter

When Flutter calls getAppConfig, Android responds with a Map containing the current session UUID and theme tokens.

private fun buildAppConfig(): Map<String, Any> {
    val uuid = getSharedPreferences("AppPrefs", MODE_PRIVATE).getString("uuid", "")
    return mapOf(
        "uuid" to uuid,
        "themeMode" to "light",
        "colors" to mapOf("primary" to "#2196F3")
    )
}

Phase 3: iOS Native Implementation

On iOS, we use SwiftUI and a FlutterManager singleton to manage the lifecycle of the Flutter engine.

FlutterManager Singleton

Similar to Android's App class, this manager handles the engine setup and MethodChannel communication. 

// FlutterManager.swift
class FlutterManager: NSObject, ObservableObject {
    static let shared = FlutterManager()
    
    func setupEngine() {
        flutterEngine = FlutterEngine(name: "esewa_flutter_engine")
        flutterEngine?.run()
        
        methodChannel = FlutterMethodChannel(
            name: "com.app.native/flutter",
            binaryMessenger: engine.binaryMessenger
        )
    }
}

Hosting the Flutter View

In SwiftUI, we warp the FlutterViewController using UIViewControllerRepresentable. This allows us to use the Flutter module as a standard SwiftUI View. 

// FlutterView.swift
struct FlutterView: UIViewControllerRepresentable {
    func makeUIViewController(context: Context) -> FlutterViewController {
        let engine = FlutterManager.shared.flutterEngine!
        return FlutterViewController(engine: engine, nibName: nil, bundle: nil)
    }
    func updateUIViewController(_ uiViewController: FlutterViewController, context: Context) {}
}

The User Interface (ContentView)

The main screen of the iOS app remains native, providing a entry point to the Flutter module via a SwiftUI sheet.

// ContentView.swift
struct ContentView: View {
    @State private var showFlutterModule = false
    
    var body: some View {
        Button("Open Flutter Module") {
            showFlutterModule = true
        }
        .sheet(isPresented: $showFlutterModule) {
            FlutterView()
        }
    }
}

Phase 4: The Communication Bridge

The MethodChannel is the glue that holds everything together. It allows for Bidirectional communication.

Direction 1: Native → Flutter (Pull)
Flutter requests config: _channel.invokeMethod('getAppConfig'). Native responds with the JSON equivalent.
Direction 2: Flutter → Native (Push)
User clicks 'Pay Now': Flutter pushes the payload: _channel.invokeMethod('initiatePayment', payload). Native receives and starts the payment process.

Implementation in Flutter

// lib/services/native_channel_service.dart
class NativeChannelService {
  static const MethodChannel _channel = MethodChannel('com.app.native/flutter');

  Future<AppConfig> getAppConfig() async {
    final Map config = await _channel.invokeMethod('getAppConfig');
    return AppConfig.fromJson(Map<String, dynamic>.from(config));
  }
}

Phase 5: 10x Security Hardening

A Senior Developer ensures that financial applications meet "Bank-Grade" security standards. Below are the hardening measures implemented.

🛡️

Encrypted Persistent Storage

Standard SharedPreferences store data in XML files that can be read on rooted devices. We've upgraded to EncryptedSharedPreferences.

// SecurityProvider.kt
fun getEncryptedPrefs(context: Context): SharedPreferences {
    val masterKeyAlias = MasterKeys.getOrCreate(MasterKeys.AES256_GCM_SPEC)
    return EncryptedSharedPreferences.create(
        "secure_prefs", masterKeyAlias, context,
        AES256_SIV, AES256_GCM
    )
}
🔒

Production Log Stripping

Logging sensitive UUIDs or API payloads in production is a major data leakage risk. We use kDebugMode to gate loggers.

// api_client.dart
if (kDebugMode) {
  _dio.interceptors.add(PrettyDioLogger(...));
}
📡

Offline Resilience (Caching)

To fulfill the optional but critical requirement for reliability, we implemented a persistence layer in Flutter.

// api_client.dart
try {
    final response = await _dio.get('/products');
    _cacheProducts(response.data);
} catch (e) {
    return await _getCachedProducts(); // Fallback
}

Architecture Summary

This tutorial flow highlights how the code blocks across three different environments (Dart, Kotlin, Swift) relate to each other:

  • State Flow: Native Auth → UUID → MethodChannel → Flutter Config → Riverpod Provider → UI.
  • UI Flow: Native Activity/View → Flutter View (hosted) → Product List → Product Detail.
  • Action Flow: Button Click (Flutter) → MethodChannel → Native Log/Payment Processor.
